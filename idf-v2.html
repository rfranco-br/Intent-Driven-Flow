01 — Why this exists

Traditional agile was built for human bandwidth. This isn't.

Scrum and Kanban are brilliant solutions to specifically human problems: memory decay, context-switching cost, estimation bias, coordination overhead. Strip out the human constraints and the rituals lose their reason. What remains is the intent behind them — shared understanding, quality protection, drift prevention, and fast feedback. IDF preserves that intent while redesigning the mechanisms for a world where agents execute and humans govern.

01

Client is the north star

Every decision at every layer must trace a clear line back to a client outcome. Features are not the unit of value — outcomes are.

02

Knowledge is distributed by design

Strategy, product, and technical truth live in separate layers held by separate people. Concentrating them creates bottlenecks and blind spots.

03

Agents execute, humans govern

Agents handle all task decomposition, building, testing, and review synthesis. Humans set direction, approve gates, and resolve ambiguity.

04

Rhythm is completion-triggered

Cycles end at the gate, not on Friday. Calendar cadence is replaced by event cadence — work flows continuously, not in batches.

05

Memory is an artifact, not a meeting

Continuity between sessions and between teams lives in maintained files, not in standup summaries or people's heads.

06

Dependencies are surfaced, not scheduled

Cross-team coordination happens when a dependency is detected, not at a weekly sync. Speed requires real-time signal flow between teams.

02 — Structure

Four layers, one client thread

The framework organises work into four concentric layers of concern. Each layer has clear ownership, distinct knowledge, and defined interfaces to the layers above and below. The client is not a layer — it is the vertical thread that runs through all of them.

Client

The client is not a stakeholder in the traditional sense — they are the reason every decision is made. Stakeholders reference client outcomes when setting strategy. The PO translates client signals into cycle intent. Executors validate that built features serve real client needs. The Guardian checks client impact before approving any gate. The client does not attend meetings or write tickets. Their presence is held by the PO and enforced by the framework.

03 — Roles

Who does what — and why

Roles are defined by the knowledge they hold, not by the tasks they perform. The critical design principle: no role holds knowledge that belongs to another layer. Developers and QAs don't disappear in IDF — they evolve. Craft Engineers and Quality Advocates are what those roles become when agents handle execution: practitioners who govern quality rather than produce output directly.

Stakeholders & Business

human

Hold the portfolio horizon, strategic intent, and investment decisions. Do not manage features or sprints — they set the direction within which product decisions are made.

OwnsTrue north, portfolio map, cross-team priorities
InterfacesPO (downward), dependency broker (cross-team)
Touch pointsNorth star sync, portfolio review, escalations

Product Owner

human

The translation layer. Holds simultaneous understanding of client needs, business constraints, and current strategic priorities. The PO's primary output is cycle intent — a clear statement of desired outcome that agents can execute against.

OwnsClient signal log, cycle intent, gate approval
InterfacesStakeholders (up), Orchestrator (down), clients (lateral)
Key disciplineWriting intent as client outcomes, not feature descriptions

Architects & Tech Leads

human

Define and maintain the technical guardrails that agents work within. They do not build features — they own the conventions, constraints, and skill files that shape how the team builds everything.

OwnsSkill library, CLAUDE.md conventions, tech constraints
InterfacesBuilder agents (via skills), Guardian (via gate criteria)
Key disciplineTranslating architectural decisions into skill files, not tickets

Orchestrator

agent

Receives cycle intent from the PO and decomposes it into a sequenced task list. Routes tasks to the correct Builder agents based on domain, tracks in-flight work, and updates project memory at cycle close.

ReadsCLAUDE.md, skill library, cycle intent statement
ProducesTask list, routing decisions, memory updates
Fails whenIntent is ambiguous — escalates to PO rather than guessing

Builder(s)

agent

Execute individual tasks. Each Builder reads the relevant skill files before starting, produces code and tests as a single unit, and operates without state between tasks. Multiple Builders can run in parallel on independent tasks within the same cycle.

ReadsTask brief, domain skill file, CLAUDE.md
ProducesCode + tests, wrapped in a feature flag by default
ConstraintStateless — no opinion carried between tasks or cycles

Guardian

agent

Reviews Builder output against the gate criteria before it reaches the PO. Runs security, performance, and cost signals. Produces a short, human-readable gate report with a clear pass / flag / fail signal. The Guardian is what makes the PO's gate review manageable.

ChecksTests pass, sec scan clean, perf budget, cost delta, client impact
ProducesGate report (≤3 paragraphs), next-cycle suggestion
EscalatesArchitectural concerns to Craft Engineer, dependency conflicts to broker

Craft Engineer

human

A developer who has shifted from writing code to governing how agents write code. Their primary output is not features — it is the quality of what the execution layer produces. They review Builder output not to catch bugs (the Guardian does that) but to catch craft issues: naming, structure, performance characteristics, and long-term maintainability that no automated check can judge.

OwnsSkill library quality, agent training, craft standards
ProducesSkill file updates, pattern escalations, craft annotations on gate reports
Also handlesGenuinely novel architectural problems and deep environmental issues agents cannot resolve
Key disciplineNoticing "the Builder keeps solving this wrong" and encoding the fix into a skill — not fixing it manually each time

Quality Advocate

human

A QA engineer who has shifted from running tests to defining what good looks like. They don't execute the test suite — the Guardian does. They decide what the test suite must verify. Before each flag goes ON, they perform a short exploratory session on the preview deploy: not scripted testing, but human judgment about whether this actually works as a real user would encounter it.

OwnsAcceptance criteria, gate criteria, exploratory validation pre-flag
ProducesGate criteria updates, quality signals to client signal log, flag-ON recommendation
CatchesFlows that are technically correct but humanly confusing — the class of problem no automated test finds
Key disciplineEncoding discovered issues into gate criteria, so the Guardian catches the same class of problem automatically next cycle

Dependency Broker

agent

Monitors cross-team signals. When a team's Planner requests something from another team, the broker detects it, notifies the relevant POs, and tracks resolution. Escalates to stakeholder level if a blocker lasts longer than one cycle.

MonitorsAll Planner outputs across teams in scope
ProducesDependency alerts, blocker log, handoff checklists
Key valueTeams don't wait — they work around while the broker resolves

Client

human

Not a workflow participant, but the ultimate measure of all decisions. The PO captures client signals continuously — feedback, usage data, support patterns — and feeds them into the client signal log. Every layer uses this log as a reality check.

ProvidesReal needs, usage signals, feedback, priority by behaviour
Represented byPO (in cycle intent), Guardian (in gate review)
Validated byFeature flag rollout, usage metrics post-deploy
04 — Rituals

Events that calibrate, not coordinate

In IDF, rituals exist to correct drift — between intent and execution, between strategy and product, between teams. They are not status updates or coordination overhead. Every ritual has a trigger, a specific output, and a clear owner. If a ritual produces no actionable output, it is skipped.

05 — Artifacts

The system's persistent memory

Artifacts are the connective tissue of the framework. Unlike Scrum's artifacts (which compensate for human memory loss), APDF artifacts are the actual memory of the system. They must be kept current — a stale artifact is worse than no artifact because it generates confidently wrong outputs.

Project memory

CLAUDE.md

The single file every agent reads at the start of every session. Contains stack, folder structure, run commands, architectural decisions, conventions, and a running log of what was built. Updated by the Orchestrator after every cycle close.

Owner: Orchestrator (writes) · Tech Lead (governs)

Skill library

.claude/skills/*.md

Domain-specific instruction sets that encode how this team builds things. Frontend components, API patterns, deploy procedures, auth conventions. Each skill is a short markdown file. Grows through skill harvests, governed by Tech Lead.

Owner: Tech Lead (governs) · Orchestrator (reads, triggers harvest)

True north document

TRUE_NORTH.md

Strategic direction and portfolio context maintained by Stakeholders. Provides the PO with the horizon against which all client signals are prioritised. Updated at North Star sync. Referenced by PO in every intent translation.

Owner: Stakeholders

Client signal log

CLIENT_SIGNALS.md

A continuously updated record of real client feedback, usage patterns, support issues, and research findings. The PO's primary input for intent translation. Referenced by the Guardian in gate reviews to evaluate client impact.

Owner: PO · Updated: continuously

Intent log

INTENT_LOG.md

A chronological record of every cycle intent statement and what was actually shipped. Backward-facing only — not a backlog. Used during drift checks to identify divergence between what was asked for and what was built over time.

Owner: Orchestrator (writes) · PO (uses in drift check)

Gate report

GATE_REPORT_{cycle}.md

Produced by the Guardian at cycle end. Three paragraphs maximum: what was built, what the signals show (tests, security, perf, cost, client impact), and a clear pass/flag/fail with a proposed next cycle. Expires when gate decision is made.

Owner: Guardian (produces) · PO (consumes)

Feature flag registry

.flags.json / flag service

All features deployed behind a flag by default. Registry tracks flag name, owner, deployment date, and current state. Enables instant rollback without revert commits. Graduates from JSON to a flag service as the product scales.

Owner: PO (toggle decision) · Builder (creates flag)

Drift register

DRIFT_REGISTER.md

A log of detected divergences between intent and implementation, with the correction taken. The most valuable long-term document in the project — it maps where the system's judgment differs from the PO's and teaches better intent writing over time.

Owner: PO · Updated: at drift checks and gate failures

Dependency registry

DEPENDENCIES.md

Cross-team contracts, shared components, and API agreements. Maintained by the dependency broker. Teams pull from it before planning a cycle to identify potential blockers early. The broker writes to it whenever a new dependency is detected.

Owner: Dependency broker · Reviewed: per dependency sync

07 — Communication patterns

How layers talk to each other

Inter-layer communication is explicit, documented, and async by default. No layer should require a synchronous meeting to receive information from another layer. The defined communication patterns below are what make fast cycles possible without creating noise or missed signals.

Strategic → Product

Direction flows down as updates to the True North document. POs read it before intent translation. No meeting required — PO is responsible for tracking changes.

Trigger: North star sync output or strategic pivot
Format: Updated TRUE_NORTH.md
Frequency: Monthly or on strategic change
Product → Execution

Cycle intent travels as a written outcome statement — not a feature list. The Orchestrator converts it to tasks. If intent is too vague, Orchestrator returns a clarification request, not a guess.

Trigger: Gate approval closes previous cycle
Format: Intent statement in INTENT_LOG.md
Frequency: Per cycle
Execution → Product

Gate reports surface from Guardian to PO. Nothing else travels upward from execution unprompted — the Guardian filters everything into a readable, actionable summary.

Trigger: All auto-tests pass, preview deployed
Format: Gate report (≤3 paragraphs)
Frequency: Per cycle
Product → Strategic

Client signals and drift register entries escalate upward when they reveal a strategic concern. PO does not escalate individual features — only patterns or conflicts that require portfolio-level decisions.

Trigger: Drift check, persistent client signal, strategic conflict
Format: Escalation note + supporting log entries
Frequency: As needed
Team A → Team B

All cross-team communication is mediated by the Dependency Broker. No direct team-to-team coordination channels — this creates noise and missed signals. The broker routes, tracks, and escalates.

Trigger: Planner detects cross-team need
Format: Dependency alert via broker
Resolution: Affected POs within 1 cycle
Client → Product

Client signals travel to the PO continuously through whatever feedback channel exists — usage analytics, support tickets, direct research. The PO consolidates them into the client signal log. This is the most important information flow in the system.

Trigger: Any client feedback, usage data, support pattern
Format: Entry in CLIENT_SIGNALS.md
Frequency: Continuous
08 — Operating principles

The rules that govern everything

A01Intent, not tasks. Humans express outcomes. Agents derive tasks. Any human writing a detailed task list is doing the Orchestrator's job and slowing the system down.
A02Client first, always. Before any gate is approved, the PO must be able to answer: "How does this serve the client?" If the answer is unclear, the gate fails regardless of technical quality.
A03Agents escalate ambiguity, they never resolve it by guessing. An agent that guesses on unclear intent will be confidently wrong. Escalation to the appropriate human is always the correct response to ambiguity.
A04Every deploy is behind a flag. No exception. This is the single practice that eliminates rollback anxiety and enables continuous deployment without continuous exposure risk.
A05The skill library is the institutional memory. If a convention isn't in a skill file, it doesn't exist for the next agent session. Knowledge that lives only in people's heads creates dependency on those people.
A06Layer boundaries are hard. The PO does not write code constraints. Tech Leads do not prioritise features. Stakeholders do not manage cycles. Crossing layer boundaries signals a broken process, not a capable person.
A07Velocity is features shipped per day, not story points per sprint. The measure of the framework's health is how frequently value reaches the client — not how accurately the team estimates effort.
A08Drift is normal. Undetected drift is failure. Every system drifts from original intent over time. The framework's job is not to prevent drift — it's to detect it early and correct it before it compounds.
A09Dependencies are surfaced in real time, not managed in weekly meetings. The dependency broker operates continuously. By the time a team is blocked, the broker has already initiated resolution.
A10The Guardian's gate report is the maximum cognitive load the PO should carry per cycle. If reviewing a cycle requires more than three paragraphs, the Guardian has failed its role, not the PO.
A11Agents amplify human craft — they don't replace it. The Craft Engineer's judgment is encoded in skill files. The Quality Advocate's judgment is encoded in gate criteria. When that judgment is absent, agents execute in a vacuum and amplify nothing. The quality of the system's output is a direct function of the quality of the human craft poured into it.
