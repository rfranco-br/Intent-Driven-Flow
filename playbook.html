W1 — Workflows

Intent Generation

Intent generation has exactly one job: produce a statement the Orchestrator can act on without asking a clarifying question. Everything else — meetings, transcripts, research synthesis, stakeholder alignment — is upstream work that feeds this moment, not part of it. When Intent Generation feels slow, the Client Signal log is almost always stale. Keep Client Signal Triage current and Intent Generation takes under an hour.

PRECONDITION
CLIENT_SIGNALS must be current before Intent Generation begins. If the signal log has not been reviewed since the last cycle or two, run Client Signal Triage first. Intent Generation starts from synthesised signals, not from a fresh meeting transcript.
01
Read the three sources
Open CLIENT_SIGNALS, TRUE_NORTH, and the current stakeholder priorities in parallel. Read for patterns, not for solutions. The signal log shows what clients are experiencing. True North shows where the organisation is pointed. Stakeholder priorities show what constraints the cycle must respect. You are looking for the intersection — an outcome that is real for the client and relevant to the current strategic direction.
02
Run AI-assisted synthesis
Load all three source documents into your AI tool and ask it to produce two or three candidate intent statements in the format: [outcome] by [mechanism], measured by [signal]. Ask the AI to cite the specific signal or True North reference driving each candidate. These are starting points, not answers. The AI surfaces patterns across large signal sets faster than a human can — but it does not know which outcome matters most right now. That judgment belongs to the PO.
03
Select and compose
Evaluate the AI candidates against one question: which outcome, if achieved, produces the most meaningful improvement for the client right now, within the current strategic horizon? Select one candidate. Edit the mechanism clause and success signal until they are precise. Do not accept AI-generated wording if it softens the outcome or introduces ambiguity.
04
Apply the template
Confirm the statement fits the canonical form.
INTENT TEMPLATE
[Client outcome] by [mechanism], measured by [signal]. Example: Reduce checkout abandonment at the payment step by presenting a single-field card input by default, measured by drop-off rate falling below 12%.
05
Anti-pattern self-check
Before logging, run one test: if you handed this statement to the Orchestrator with no additional explanation, could it begin decomposition without asking a clarifying question? If yes, the intent is ready. If no, the mechanism clause is either too vague (Orchestrator cannot determine scope) or too specific (Orchestrator is being told how, not what). Tighten and re-check.
EPIC-FRAMEDDescribes a feature, not an outcome. No success signal possible.
STORY-FRAMEDTask disguised as intent. Orchestrator will build exactly this, nothing more.
SPEC-FRAMEDImplementation detail. PO has already done decomposition. Agents won't question it.
06
Log and trigger
Append the intent entry to INTENT_LOG and trigger the Orchestrator. The log entry is the only artifact Intent Generation produces. Working notes, AI-generated candidates, and draft iterations are ephemeral — they do not enter the framework artifact set.
INTENT_LOG — new entry
## Cycle [N] — Intent Date: YYYY-MM-DD Owner: Product Owner Intent: [outcome] by [mechanism], measured by [signal]. Signal source: [CLIENT_SIGNALS ref] — one sentence of what it showed. True North alignment: [One sentence confirming fit, or flag a tension.] Stakeholder priority: [Which priority this serves, or "none explicitly — justified by signal weight".] Orchestrator notes: [Optional. Context that prevents Orchestrator escalation. Not a task list. If the mechanism involves a technically novel approach, name it here so the Architect sees it during decomposition.]
INTENT GENERATION DOES NOT COVER
— How to run a client meeting — outside IDF scope
— How to triage raw signals into CLIENT_SIGNALS — see Client Signal Triage
— How to write the intent statement itself — see Writing Intent Statements
— Arch or Tech Lead approval of intent — the correct IDF pattern is an Orchestrator notes field in the log entry, not a synchronous gate
GREENFIELD — NO SIGNALS YET
When CLIENT_SIGNALS does not exist or is empty, intent cannot be derived from observed client behaviour. Write an assumption-based intent instead. The format is identical — [outcome] by [mechanism], measured by [signal] — but the signal is a hypothesis validation target, not a performance metric. Mark the INTENT_LOG entry with Signal source: assumption — [one sentence stating what you believe to be true about the client's need]. The cycle that follows is a learning cycle: its purpose is to validate or invalidate the assumption, not to deliver a known improvement. If the assumption is invalidated, the Escape Hatch applies and the learning is logged.
CONSTRAINED GREENFIELD — KNOWN RESTRICTIONS, NO SIGNALS
When building new work that must later integrate with an existing corporate or platform ecosystem, predefined constraints (integration standards, auth requirements, infrastructure dependencies) belong in System Memory before the first cycle begins — not in the intent statement. The Orchestrator reads System Memory automatically. The PO does not restate constraints per cycle. Use the Orchestrator notes: field in the INTENT_LOG entry only for constraints that are not yet in System Memory and are novel to this specific cycle. Signals may be absent at the start; constraints must not be.
W2 — Workflows

Client Signal Triage

Client Signal Triage has one job: convert raw observations into structured CLIENT_SIGNALS entries the PO can read and act on. Raw signals are noise — support tickets, interview transcripts, analytics alerts, sales call notes. A structured signal is a pattern with a severity, a frequency estimate, and a direct line to a client outcome. Triage is the work that turns the first into the second.

WHEN TO RUN CLIENT SIGNAL TRIAGE
Run Client Signal Triage before Intent Generation begins, whenever the signal log has not been reviewed since the last one or two cycles. The trigger is not a schedule — it is the PO noticing that the CLIENT_SIGNALS log no longer reflects what clients are experiencing right now. A log that is more than two cycles out of date is context debt: Intent Generation running on stale signals will produce cycle intent that was accurate last month, not today.
01
Collect the raw inputs
Pull together everything that arrived since the last triage: support tickets (especially recurring ones), direct client feedback, sales and CS call notes, analytics anomalies, and any in-product behavioural data you have access to. Do not filter yet — collect first. The goal at this step is a single pile, not a prioritised list. If your team uses a CRM, helpdesk, or analytics tool, export a summary rather than reading every item individually.
02
Identify patterns, not incidents
A single support ticket is an incident. Three tickets describing the same friction in different words are a pattern. Signal triage looks for patterns. Read the pile for recurrence: the same step failing for different users, the same feature absent in multiple feedback threads, the same question appearing repeatedly in client calls. An incident that appears once belongs in a notes file, not in CLIENT_SIGNALS. A pattern that appears three or more times — or once with a high-severity, business-critical impact — earns a structured entry.
03
Score each pattern
Before writing the entry, score the pattern on two dimensions: frequency (how many clients are affected, roughly) and severity (what happens to the client if this is never fixed — minor friction, blocked workflow, or lost value). High frequency + high severity entries go to the top of the log and feed the next Intent Generation session. Low frequency + low severity entries go in, but are marked low-priority. Do not discard them — patterns that are low-priority today become high-priority when a related signal clusters around them next cycle.
Frequency
Severity
HIGH — reported by 3+ distinct users or clients in this period
HIGH — client is blocked, loses value, or churns without this resolved
LOW — seen once or twice; may be edge case
LOW — minor friction; workaround available; does not block core task
04
Write the structured entry
Each signal entry describes the pattern in client terms — what the client experiences, not what the system does wrong. The entry is not a bug report or a feature request: it is an observation of client reality. The PO uses it to write intent; the intent should be traceable to at least one entry in this log.
CLIENT_SIGNALS — entry format
## [Short pattern name] Date added: YYYY-MM-DD Frequency: [HIGH / LOW] — [brief note, e.g. "5 tickets in last 2 weeks"] Severity: [HIGH / LOW] — [brief note, e.g. "blocks checkout completion"] Source(s): [Where this came from — support, interview, analytics, etc.] Observation: [2–3 sentences describing what clients experience. Write in client terms. "Users cannot complete payment when..." not "The payment endpoint returns 500..."] Intent candidate: [Optional. One sentence in intent format: [outcome] by [mechanism]. Leave blank if the pattern is not yet clear enough to generate intent.] Status: [ACTIVE / ADDRESSED — Cycle [number] / ARCHIVED]
05
Update existing entries
Do not add a new entry for every triage session. If the pattern already exists in the log, update the frequency count, add a new source reference, and refresh the date. A pattern that is growing across cycles signals an unresolved client need — that escalation in frequency is itself valuable signal for the PO. An entry marked ADDRESSED should be verified: if new signals on the same pattern appear after a cycle closed, reopen it rather than creating a duplicate.
06
Hand off to Intent Generation
When triage is complete, the log should contain: all active patterns, ordered roughly by frequency × severity, with at least one HIGH/HIGH entry that is ACTIVE. That entry is almost always the input to the next Intent Generation session. Signal the PO that the log is current. Triage is complete when the PO can open CLIENT_SIGNALS and identify the most pressing client need in under two minutes.
CLIENT SIGNAL TRIAGE DOES NOT COVER
— How to run a client interview or feedback session — outside IDF scope
— How to set up analytics tooling or event tracking — outside IDF scope
— How to write the final intent statement — see Writing Intent Statements
— Prioritisation between competing intent candidates — that is the PO's judgment in Intent Generation, using True North and stakeholder priorities as inputs
AI-ASSISTED TRIAGE
AI tools can accelerate steps 01–03 significantly. Load your raw inputs — ticket export, call notes, feedback thread — and ask the AI to identify recurring patterns, group by theme, and produce a draft signal entry for each pattern above a frequency threshold you define. Review every draft before it enters the log. The AI identifies recurrence fast; it does not understand which patterns actually block client outcomes versus which merely produce complaints. That distinction belongs to the PO.

W4 — Workflows

Gate Review Decision

Gate Review Decision has one job: give the Product Owner a clear, repeatable procedure for reading a Guardian gate report and recording a PASS, FLAG, or FAIL decision. The Guardian produces the evidence. The PO makes the call. No agent makes this decision, no engineer overrides it, and no cycle ships without it.

PRECONDITION
The Guardian has run and produced a Gate Report. The Gate Report is linked or attached to the cycle context. The PO has the original Intent Statement for this cycle available. If any of these is missing, do not begin — return to the Guardian first.
WHEN TO RUN GATE REVIEW DECISION
Run Gate Review Decision at the end of every build cycle, after the Guardian signals build complete and produces a gate report — whether that report is generated automatically or prompted by the Orchestrator. Never skip this step. No build ships without a PO gate decision, regardless of test status or team confidence.
01
Open the Gate Report
Locate the latest Gate Report artifact for this cycle. Confirm it was produced by the Guardian — not manually authored by a Builder or Orchestrator. The report must follow the three-paragraph structure: verdict, evidence, next cycle. If no report exists, or if the report does not follow the standard structure, stop and return to the Guardian. A non-standard or absent report is not a reason to proceed — it is a precondition failure.
02
Read the verdict paragraph
Read the first paragraph. The Guardian has already written a recommended verdict — PASS, FLAG, or FAIL — with one sentence of primary reasoning. Note it, but do not adopt it automatically. The PO's job is to verify the verdict is supported by the evidence, not to ratify the Guardian's judgment. The Guardian may recommend FLAG on a borderline case that the PO decides to PASS, or vice versa. The PO's decision is the gate — the Guardian's verdict is input to it.
03
Check the evidence paragraph
Read the second paragraph. Verify every evidence line is present: test counts, security result, performance delta, cost delta, and client impact scope. If any line is absent — not "clean" or "no change", but genuinely missing — treat it as a failed check. A missing line means the check was skipped or the Guardian did not have the data; it does not mean the check passed. Ask: does the evidence support the verdict? If tests passed but performance is outside budget, the verdict should be FLAG, not PASS. If the Guardian's verdict conflicts with the evidence, the PO decides — not the Guardian.
04
Read the next-cycle paragraph
Read the third paragraph. This is the Guardian's observational record — patterns noticed during the cycle, dependencies that surfaced, risks worth watching. These observations are non-binding: the PO decides which, if any, to act on. Items that belong here: new technical debt introduced, a dependency risk that emerged, a partial implementation deferred to the next cycle. Items that do not belong here: task lists, feature requests, unilateral recommendations to delay shipping. If the next-cycle paragraph contains task assignments or overrides the shipping decision, flag that as a Guardian anti-pattern for a future cycle.
05
Check against the cycle intent
Open the original Intent Statement for this cycle. Ask two questions. First: does what was built match what the intent said — correct outcome, correct mechanism constraint respected? If the Orchestrator escalated mid-cycle and the scope was adjusted, confirm the adjusted scope is still within the original intent boundary and that the adjustment was logged. Second: does what shipped move toward or away from True North? A build that passes all technical checks but drifts away from the strategic direction is still a FLAG. True North alignment is a gate criterion, not a nice-to-have.
06
Make the gate decision
Choose one of three decisions. The criteria are:
Decision
Condition
Next action
PASS
Intent met. All checks green. No True North conflict. No unresolved risk in observations.
Flag ON. Cycle closed. Notify team.
FLAG
Intent partially met, or one or more checks failed, or an observation risk is too high to ship without a follow-up commitment.
Flag stays OFF. Write a follow-up intent for the next cycle. Do not reopen this cycle.
FAIL
Intent not met, or quality gate failed with no path to a quick fix, or the build drifts from True North in a way that cannot be resolved by a follow-up intent alone.
Flag stays OFF. Cycle closed without shipping. Write retrospective note. Rewrite intent for next cycle.
07
Record the decision
Update the Gate Report with the PO decision and a one-sentence rationale. This is not a detailed post-mortem — it is a one-line record that closes the cycle: who decided, what they decided, and the primary reason. This record becomes part of the permanent cycle history and is the input a future PO or team member uses to understand why a flag was or was not turned on.
GATE REPORT — PO decision record
PO Decision: [PASS / FLAG / FAIL] Date: YYYY-MM-DD Rationale: [One sentence. E.g. "FLAG — performance outside budget for 8% of users; follow-up intent written for next cycle."]
08
Hand off
Act on the decision immediately.

PASS: notify the team the flag is live. The cycle is closed. The next cycle begins when a new intent is written.

FLAG: write the follow-up intent before the next cycle begins. The follow-up intent is a new, complete intent — not an amendment to the current one. The current cycle's flag stays OFF until the follow-up cycle passes gate.

FAIL: write a retrospective note to the INTENT_LOG entry for this cycle, stating what was not met and why. Reset intent for a full new cycle. Do not carry the failed intent forward unchanged — a failed cycle is signal that the intent, the mechanism, or the measurement was wrong. The retrospective note is what makes the next cycle smarter.
GATE REVIEW DECISION DOES NOT COVER
— How to write or produce the gate report — see Gate Report (Artifact Templates)
— How to fix failing checks — that is Builder and Guardian work, resolved before the next gate
— How to rewrite a failed or flagged intent — see Intent Generation
— Whether to run a cycle at all — that is Intent Generation
— How to turn a feature flag on in the codebase — that is a Builder task triggered by PASS
W5 — Workflows

Context Reset

Context Reset has one job: audit every persistent artifact for staleness, conflicts, and decay before the next cycle begins. It is not a retrospective. It is not a planning meeting. It is a targeted artifact hygiene operation — the step that ensures what agents read is still accurate before they execute. Stale context corrupts agent decisions. A Context Reset is how the team prevents that corruption from compounding across cycles.

PRECONDITION
A staleness trigger has fired — see System Memory (Artifact Templates) for the trigger list. The PO has approved the reset; it delays the next cycle and must not begin without PO sign-off. All artifacts in scope are accessible — no artifact is locked or currently being updated by another team member.
WHEN TO RUN CONTEXT RESET
— After any cycle that ends in FAIL
— When System Memory has not been updated in 3 or more cycles
— When a role changed hands
— When the tech stack changed materially
— When agents begin producing unexpected output patterns
— Proactively every 5–7 cycles even if no trigger fires — artifact decay is gradual; waiting for a visible trigger means some decay has already affected work
01
Confirm scope
PO and Craft Engineer agree on which artifacts are in scope for this reset. The default scope is: System Memory, Capability Library, Client Signal Log. Optional scope: Feature Governance Registry, Drift Register, TRUE_NORTH. Scope should match the trigger — if only System Memory is stale, a full five-artifact audit is waste. If a role changed hands, all artifacts are in scope because every artifact reflects assumptions made by the previous role-holder. Agree and write down the scope before any audit begins.
02
Audit System Memory
Read System Memory top to bottom. Mark every entry that is: outdated (the reality it describes has changed), conflicted (contradicts another entry in the file or another artifact), or orphaned (refers to a decision, feature, or system that no longer exists). Remove or update each marked entry. Do not archive stale entries — stale System Memory should be deleted, not moved. A stale entry that is merely archived to a notes file still exists and may still be read. Update the "Last reviewed" field at the top of the file when the audit is complete.
03
Audit the Capability Library
Review every capability file in scope. For each file, ask: does this file still describe what the Builder currently does? Are the examples current? Does this file describe a capability that was deprecated or replaced? Flag outdated files to the Craft Engineer for rewrite. Do not leave a file that describes the old behavior — an agent reading it will follow the pattern it describes, not the pattern the team currently uses. If a capability no longer exists, delete the file rather than marking it deprecated in place.
04
Audit the Client Signal Log
Scan for entries with ACTIVE status older than three cycles. For each one, make a decision: still valid (leave ACTIVE, confirm it with a date refresh), promoted to cycle intent (change status to ADDRESSED with a cycle reference), or no longer relevant (change status to ARCHIVED with a one-sentence reason). The signal log should not be a graveyard of stale ACTIVE signals. An ACTIVE signal the PO no longer believes in is context that biases the next Intent Generation session. Resolve or archive it.
05
Audit optional artifacts (if in scope)
If the Feature Governance Registry is in scope: remove flags that are permanently ON — deployed, stable, and no longer under governance. They are shipped features, not governance objects, and continuing to list them as governance objects adds noise to every future review. If the Drift Register is in scope: review every open drift entry. Has it been resolved? If yes, close it with a resolution date. If not, it is an unresolved input to the next cycle's intent. An open drift entry that is not feeding the next cycle's intent is stale by definition.
06
Resolve conflicts
If any two artifacts contradict each other — for example, System Memory describes one tech constraint while a Capability Library file assumes a different one — resolve the conflict before the reset is complete. The resolution goes into the artifact with the broader scope, which is usually System Memory. Do not leave the conflict in place with a note that it "needs discussion later." An agent reading conflicting artifacts during the next cycle will choose one arbitrarily or escalate. Resolve it now, and log the resolution decision briefly in the artifact where the change was made.
07
Signal reset complete
Craft Engineer confirms to PO: all in-scope artifacts audited, no known conflicts, team ready to execute the next cycle. PO signs off. The next cycle's intent injection begins from the reset state — agents receive the updated context as their starting state, not the pre-reset state. Before sign-off, run the post-reset checklist.
POST-RESET CHECKLIST
□ System Memory: last-reviewed date reflects this reset □ Capability Library: no files describing deprecated behavior □ Client Signal Log: no ACTIVE entries older than 3 cycles without review □ Feature Governance Registry (if audited): no permanently ON flags still listed □ Drift Register (if audited): all resolved entries closed □ Conflicts: none unresolved □ PO sign-off: confirmed
CONTEXT RESET DOES NOT COVER
— How to update individual System Memory entries — see System Memory (Artifact Templates)
— How to write a new Capability Library file — see Writing Capability Library Entries
— How to triage new client signals — see Client Signal Triage
— Whether a cycle failed — that is Gate Review Decision; Context Reset runs after the gate decision, not instead of it
— How to log intent corrections mid-cycle — see Drift Register (Artifact Templates)

T3 — Artifact Templates

GATE_REPORT

The gate report is what the Guardian produces at the end of every cycle. It is the only artifact the PO reads before making a pass, flag, or fail decision. Its job is to compress an entire cycle's output into a scannable summary — not to document everything that happened, but to surface exactly what the PO needs to decide. A gate report that requires the PO to dig further has failed its purpose.

The three-paragraph structure

Every gate report has exactly three paragraphs, in this order. The Guardian does not add sections, summaries, or appendices. The PO reads all three in under two minutes.

Paragraph 1
Verdict

One sentence. PASS, FLAG, or FAIL — followed by one sentence stating the single most important reason. The PO reads this and knows immediately what decision is in front of them.

Paragraph 2
Evidence

Three to five bullet points. Tests passed or failed (with counts), security scan result, performance delta against budget, cost delta, and one sentence on client-impact scope. Facts only — no interpretation. The PO uses this to verify the verdict is supported.

Paragraph 3
Next cycle

One to two sentences. What the Guardian observed that is worth acting on in the next cycle — a pattern in test failures, a dependency that surfaced, a performance trend. Not a task list. The PO decides whether to act on it.

The template

GATE_REPORT — structure
## Gate Report — Cycle [N] Date: YYYY-MM-DD Intent: [One-line restatement of the cycle intent] Flag: [FLAG_NAME] --- [PASS / FLAG / FAIL] — [One sentence: the primary reason for this verdict.] --- - Tests: [N passed / N failed — name any failing tests] - Security: [clean / [finding description]] - Performance: [delta vs budget — e.g. "p95 latency +12ms, within 50ms budget"] - Cost: [delta — e.g. "estimated +$0.004/request, within $0.01 budget"] - Client impact: [Who is affected and how, if flag is turned ON] --- [One or two sentences on what to watch or act on next cycle. If nothing notable: "No escalations. Proceed at normal cadence."]

Annotated examples

PASS example
## Gate Report — Cycle 14 Date: 2026-04-01 Intent: Reduce checkout abandonment at the payment step by presenting a single-field card input by default, measured by drop-off rate falling below 12%. Flag: CHECKOUT_SIMPLIFIED_INPUT --- PASS — All gate criteria met; the simplified input renders correctly across all tested viewports and the drop-off measurement hook is in place. --- - Tests: 47 passed / 0 failed - Security: clean - Performance: p95 latency +8ms on checkout load, within 50ms budget - Cost: no change (client-side only) - Client impact: affects all users at the payment step; flag is scoped to the card input field only — no other checkout behaviour changes --- No escalations. The existing autofill test suite covers this change adequately. Proceed at normal cadence.
WHY IT WORKS — Verdict is immediate. Evidence is factual and complete. The "no change" cost line is explicit rather than absent — absence looks like an oversight. Next-cycle note is minimal because nothing needs acting on.
FLAG example — PO must decide
## Gate Report — Cycle 15 Date: 2026-04-02 Intent: Allow users to export their order history as a CSV file, measured by export feature used by 15% of returning users within 14 days. Flag: ORDER_HISTORY_EXPORT --- FLAG — Tests pass and the export produces valid CSV, but the generation time for accounts with 500+ orders exceeds the 3-second UX threshold. --- - Tests: 31 passed / 0 failed - Security: clean — no PII logged; export scoped to requesting user only - Performance: export generation at p95: 2.1s (under 500 orders), 6.8s (500+ orders) — exceeds 3s budget for ~8% of user base - Cost: +$0.002/export request, within $0.01 budget - Client impact: affects returning users with order history; 8% of returning users have 500+ orders --- Recommend: PO to decide whether to release behind a staged rollout (excluding accounts with 500+ orders initially) or hold for a performance fix. Guardian has escalated the generation pattern to the Craft Engineer for review.
WHY IT WORKS — Flag verdict explains the specific condition that triggered it. The evidence quantifies the problem precisely (6.8s, 8% of users) so the PO can make a risk decision. The next-cycle paragraph offers a concrete option without prescribing the answer — that judgment belongs to the PO.
FAIL example
## Gate Report — Cycle 16 Date: 2026-04-03 Intent: Prevent duplicate order submissions on double-tap by adding server-side idempotency on the order creation endpoint, measured by duplicate order rate falling to zero in the next 48-hour window. Flag: ORDER_IDEMPOTENCY --- FAIL — 3 tests failing in the idempotency suite; duplicate orders are still created under concurrent request conditions. --- - Tests: 28 passed / 3 failed (order_idempotency_concurrent_post × 2, order_idempotency_race_condition × 1) - Security: clean - Performance: no regression - Cost: no change - Client impact: flag must remain OFF; the failure condition produces real duplicate charges --- The race condition occurs when two requests arrive within the same database transaction window. The idempotency key is being checked before the first write commits. Escalated to Craft Engineer — this requires a transaction-level fix, not an application-level key check.
WHY IT WORKS — Fail verdict names the specific tests and why they matter (real duplicate charges). The escalation in the next-cycle paragraph gives the Craft Engineer exactly what they need to diagnose — no additional investigation required. The flag stays OFF because the evidence makes the risk clear.
GATE REPORT ANTI-PATTERNS
— Verdict buried in paragraph two. The PO should not have to read to find the decision. Verdict is always first.
— Missing lines in the evidence paragraph. Absent security or performance lines look like the check was skipped, not that nothing changed. State the result even if it is "clean" or "no change."
— Next-cycle paragraph that is a task list. The Guardian observes and recommends. It does not assign tasks or set priorities. Those decisions belong to the PO.
— FLAG with no decision criteria. A flag without a concrete condition for the PO to evaluate forces a judgment call with no anchor. Every flag must include what the PO is deciding.
T4 — Artifact Templates

System Memory

System Memory is the single file every agent reads before acting. It holds the team's decisions, constraints, and conventions — the context that would otherwise have to be re-explained every cycle. A well-maintained System Memory file is what makes agents consistent. A stale or empty one is the most common cause of cycles that produce correct-but-wrong output.

WHAT SYSTEM MEMORY IS NOT
System Memory is not a project wiki, a meeting notes dump, or a technical design document. It contains only the information agents need to act correctly — decisions, constraints, and conventions that apply across cycles. If a piece of information would not change how an agent behaves, it does not belong here.

The starter template

Copy this structure when starting a new team. Fill the required sections before the first cycle. Leave optional sections empty rather than filling them with placeholder text — agents treat empty sections as absent, not as gaps to fill.

SYSTEM_MEMORY — starter file
# System Memory Last reviewed: YYYY-MM-DD Owner: [Role responsible for keeping this current] --- ## Product context [One paragraph. What does this product do, who uses it, and what does success look like for a typical user? Write this for an agent that has never seen the product.] --- ## Technical constraints [Bullet list. Hard rules agents must never violate. Examples: auth method, deployment target, language version, infrastructure limits, third-party integration requirements. Do not list preferences here — only constraints.] --- ## Architecture decisions [Bullet list. Decisions that have been made and closed. Not the reasoning — just the decision and its scope. Example: "All API routes use REST. GraphQL not in scope." If a decision can be reopened by an agent, it is not ready for this section.] --- ## Conventions [Bullet list. Naming patterns, file structure rules, commit message format, branch naming, test patterns. Agents use this section to self-check output before submitting. Be specific enough that "does this follow our conventions?" has a binary answer.] --- ## PO gate authority Backup delegate: [Name] If PO is unavailable: [Name holds gate authority] Escalation: [How to reach the PO delegate if a cycle is blocked] --- ## Active feature flags [List of flags currently in Pending-OFF or Live-ON state. Format: FLAG_NAME — description — current state — date set Remove Dead-OFF flags promptly. This list is agent context.] --- ## Known dependencies [External services, teams, or systems the agents must be aware of. Include SLA, contact, and any known failure modes.] --- ## Off-limits [Explicit list of things agents must not touch, modify, or make assumptions about. Examples: production data, third-party credentials, legacy systems under migration.]

Section guide

Section
Purpose and rules
Product context
Required. The orientation paragraph. Agents use it to understand scope when decomposing intent. Without it, agents may produce technically correct output that misses the product's core purpose. Keep it to one paragraph — if it needs more, it is too broad to be useful.
Technical constraints
Required. Hard limits, not preferences. A constraint is something that, if violated, produces a broken or non-deployable system. Preferences belong in Conventions. If a constraint has an expiry (e.g. a migration in progress), note the date — stale constraints are context debt.
Architecture decisions
Required once made. Closed decisions only. An agent that reads an open question in this section may attempt to resolve it. Each entry should read as: what was decided, and what is out of scope as a result. Do not include the debate — only the outcome.
Conventions
Required. The list that makes agent output consistent across cycles. Agents self-check against this before producing output. The more specific the entries, the less craft review overhead. Vague conventions ("write clean code") produce nothing. Specific ones ("all async functions must have an error boundary") are enforced.
PO gate authority
Required. IDF requires the PO to be the sole gate authority. Without a named backup, an unavailable PO holds the entire cycle at the Deploy step indefinitely. This section must be current — an outdated delegate is equivalent to no delegate.
Active feature flags
Required — living. Agents use this list to understand what is currently live, what is pending release, and what is safe to touch. Dead-OFF flags left here become context pollution — stale context corrupts future agent decisions. Remove them promptly. The Craft Engineer is responsible for flag hygiene.
Known dependencies
Required once known. Agents discovering a dependency for the first time during decomposition will escalate or guess. This section prevents both. Include the dependency's reliability characteristics — an agent that knows a service has a known latency issue will plan differently.
Off-limits
Required when applicable. The explicit boundary list. Agents do not intuit that certain systems are untouchable — they need to be told. A missing entry here is how migrations get disrupted and production data gets touched during a routine cycle.

Entries that prevent the most common early failures

These are the specific entries new teams most often omit — and pay for within the first three cycles.

MISSING: AUTH PATTERN
The most frequently re-implemented thing in any codebase. Without it in System Memory, every Builder agent re-derives auth from context — producing inconsistent session handling, token storage, and middleware across cycles. Add the auth method, token storage location, and session timeout rule on day one.
## Technical constraints - Auth: JWT, stored in httpOnly cookie, 24h expiry - Session refresh: sliding window, max 7 days - Route protection: all /api/* routes require valid token
MISSING: ERROR HANDLING CONVENTION
Agents produce error handling that is technically correct but inconsistent — some throw, some return error objects, some log silently. A single convention entry eliminates this class of craft review feedback across all future cycles.
## Conventions - Error handling: all async operations use try/catch; errors logged to [logging service] with context object; never silently swallowed - API errors: return { error: string, code: string }; never expose stack traces to client
MISSING: TEST SCOPE BOUNDARY
Without a clear test scope entry, agents write tests for everything they can reach — including third-party integrations, external APIs, and infrastructure. This inflates test suites, increases false-positive failure rates, and slows gate reviews. Define the boundary explicitly.
## Conventions - Unit tests: required for all business logic functions - Integration tests: required for all DB write operations - Do not test: third-party SDK internals, external APIs, infrastructure provisioning - Test file location: co-located with source (*.test.ts)
MISSING: MIGRATION / LEGACY SYSTEM BOUNDARY
If any part of the system is under migration, being deprecated, or must not be modified during the migration window, it must be in Off-limits. An agent with no knowledge of a migration in progress will make changes to the system being replaced — because from its context, those changes are correct.
## Off-limits - /legacy-api/* — under migration to v2, do not modify (migration owner: [Name], expected completion: YYYY-MM-DD) - users_v1 table — read-only during migration window; all writes go to users_v2
STALENESS SIGNALS
System Memory becomes context debt when it describes a system that no longer exists. Watch for these signals:

— Agents produce output that was correct three cycles ago but wrong now
— Craft review finds the same category of issue in consecutive cycles
— A PO gate question reveals an architectural decision not in System Memory
— A cycle touches an "off-limits" area without triggering an escalation

Any of these is a trigger for a Context Reset — see Context Reset. The Last reviewed date at the top of the file must be updated every time the file is edited — not on a schedule.

C1 — Communication Patterns

Writing Intent Statements

The intent statement is the only input the Orchestrator receives from a human. Everything the system produces in a cycle flows from it. A statement that is too vague produces a cycle that drifts before the first task is complete. A statement that is too specific means the PO has already done the Orchestrator's job — and the agents will build exactly what was specified, no more, no less, even if it is wrong.

The canonical format

INTENT TEMPLATE
[Client outcome] by [mechanism], measured by [signal]. Example: Reduce checkout abandonment at the payment step by presenting a single-field card input by default, measured by drop-off rate falling below 12%.

Three clauses. Each does a distinct job:

CLIENT OUTCOME

What changes for the client if this cycle succeeds. Not what the team will build — what the client will experience differently. This is the only clause the PO fully controls. The others are aids to decomposition.

MECHANISM

The approach the team believes will produce the outcome. This constrains the Orchestrator's decomposition without specifying implementation. It must be specific enough to prevent the Orchestrator from asking a scoping question, but not so specific that it dictates how to build.

SUCCESS SIGNAL

How the PO will know the outcome was achieved — a measurable client-side indicator. This is what makes a gate report binary: pass if the signal moved, fail if it did not. Without this clause, the gate review becomes a judgment call with no anchor.

Scope: is this intent cycle-sized?

Format is necessary but not sufficient. An intent can be perfectly formatted and still be too large to complete in a single cycle. The test is not word count or complexity — it is whether a single gate report can confirm the outcome was achieved.

THE GATE TEST
Could the PO read a single gate report and make a binary pass/fail decision on this outcome? If yes, the intent is cycle-sized. If no, the intent spans multiple independently testable outcomes and must be decomposed by the PO before the cycle begins.
CYCLE-SIZED

"Allow users to set their profile to invite-only, measured by the feature being used by 20% of active users within 7 days."

One flag. One gate condition. The PO can pass or fail this in one review.

TOO LARGE

"Build a social network where users control who joins, measured by user satisfaction."

No single gate can confirm this. The outcome requires multiple flags, multiple releases, and months of signal accumulation. This is a product vision, not an intent.

The under-specified / over-specified spectrum

Most intent failures land at one of two extremes. Both produce bad cycles for different reasons.

Type
What it looks like
What breaks
UNDER-SPECIFIED
"Improve the user profile." No mechanism, no signal. The Orchestrator must invent scope.
Orchestrator escalates immediately — ambiguity halts execution. If it doesn't, it is making product decisions on behalf of the PO.
OVER-SPECIFIED
"Add an 80×80px circular avatar field to the profile edit form, stored in S3." Implementation detail masquerading as intent.
Agents build exactly this. No more. If the spec is wrong, the output is wrong — and the gate report will pass because the spec was met, not because the client outcome was achieved.
WELL-FORMED
"Let users update their profile picture by uploading an image from their device, measured by profile completion rate increasing by 10%."
Outcome is clear. Mechanism constrains scope without dictating implementation. Signal makes the gate binary.

Self-check before logging

01Could the Orchestrator begin decomposition without asking a clarifying question? If no — the mechanism clause is too vague.
02Could a single gate report confirm the outcome was achieved? If no — the intent is too large. Decompose before logging.
03Does the success signal measure a client-side change? If it measures internal activity (deploys, tickets closed, hours worked) — rewrite it.
04Can you trace this intent to at least one source in CLIENT_SIGNALS, True North, or stakeholder priorities? If no — do not log it yet.
05Would this intent require more than one feature flag to release safely? If yes — it is more than one cycle. Split it.

P2 — Team Patterns

Running Your First Cycle

The first cycle is not a pilot — it is the real system running at reduced scope. The goal is to close a complete loop: intent → decomposition → build → gate → flag ON. Everything that prevents that loop from closing is a setup problem, not a framework problem. Fix it before cycle two.

BEFORE YOU START
The first cycle must be small enough that failure is cheap. Choose an intent that is genuinely useful but isolated — something that touches one area of the codebase, has a clear success signal, and does not block other work if it takes longer than expected. The first cycle teaches the team how the system works. It is not the place to prove the framework on a complex problem.

Phase 1 — Before the cycle begins

Complete these in order. Do not start the cycle until all four are done.

01
Assign roles — explicitly
Name who holds each role for this cycle. For a small team, one person may hold multiple human roles — but the Product Owner and the Craft Engineer must be separate people if possible. The PO holds gate authority; the Craft Engineer reviews agent output. If the same person does both, there is no quality check before the gate. Write the assignments into System Memory before the cycle starts.
System Memory — role assignments
## Role assignments — Cycle 1 Product Owner: [Name] PO backup delegate: [Name] Craft Engineer: [Name] Quality Advocate: [Name or "PO acting" if solo] Architects / TLs: [Name or "none designated yet"]
02
Bootstrap System Memory
Use the System Memory template. Fill Product context, Technical constraints, Conventions, and PO gate authority before anything else. Leave Active feature flags and Known dependencies empty — they will populate from the first cycle. Do not fill sections with placeholder text. An empty section is honest. A placeholder is noise the agent will try to interpret.
MINIMUM VIABLE SYSTEM MEMORY
Product context paragraph + Technical constraints list + three Conventions + PO gate authority. That is enough to run the first cycle. Everything else can be added as the team discovers it.
03
Create the minimum artifact set
The first cycle needs three files to exist before the Orchestrator can start: System Memory, INTENT_LOG, and a feature governance registry (even if empty). CLIENT_SIGNALS should exist but can start with a single entry synthesised from what the team already knows about the client's most pressing need. TRUE_NORTH is optional for cycle one — but if it is absent, note that fact in the INTENT_LOG entry so the Orchestrator knows the intent was not validated against a strategic direction document.
Minimum artifact checklist
[ ] SYSTEM_MEMORY — Product context, constraints, conventions, gate authority [ ] INTENT_LOG — empty file ready to receive first entry [ ] FEATURE_GOVERNANCE_REGISTRY — empty file (flags added by Builder) [ ] CLIENT_SIGNALS — at least one entry before Intent Generation begins [ ] TRUE_NORTH — optional; note absence in INTENT_LOG if missing
04
Write the first intent
Use the Intent Generation workflow and Writing Intent Statements. The first intent should be deliberately narrow. The team is learning the system — not just what to build, but how the Orchestrator decomposes, how the Builder produces output, and what the Guardian flags. A narrow intent produces a short cycle with a fast feedback loop. A broad intent on cycle one will produce a long cycle where every learning is delayed.
GOOD FIRST INTENT
One feature flag. Touches one area of the codebase. Success signal is observable within 24 hours of flag ON. The Orchestrator can decompose it in under 5 tasks.
AVOID FOR CYCLE ONE
Auth system changes, database migrations, cross-service integrations, or anything that would require multiple flags or block other work if it stalls.

Phase 2 — During the cycle

What to watch, what to resist, and when to intervene.

Stage
What to watch for
Orchestrator decomposition
If the Orchestrator escalates immediately, the intent is ambiguous — ambiguity halts execution (see Operating Principles). Do not push through — rewrite the intent. If the Orchestrator produces a task list that looks like a feature spec, the mechanism clause was too detailed. Note this for Writing Intent Statements refinement after the cycle.
Builder execution
Resist reviewing Builder output before the Guardian runs. First-cycle instinct is to check every task as it completes — this creates a shadow review loop that bypasses the gate. Let the cycle run. The Guardian report will tell you what to inspect.
Guardian gate
A FLAG or FAIL on cycle one is useful. It means the gate criteria are working. Read the report carefully — the specific failure tells you what to add to System Memory or Capability Library before cycle two. A PASS on cycle one with thin gate criteria is a warning sign, not a success.
PO gate review
The PO reads the gate report and makes a binary decision. If the PO finds themselves wanting more information before deciding, the gate report structure is insufficient — note what was missing for Gate Report template refinement. If the PO overrides a Guardian FLAG, document the reason in the INTENT_LOG.

Phase 3 — After the cycle closes

01
Update System Memory
Add anything the Orchestrator or Builder discovered that is not yet in System Memory — new dependencies encountered, architectural decisions surfaced during decomposition, conventions violated that need to be made explicit. Update the "Last reviewed" date.
02
Update the Capability Library
If the Craft Engineer found patterns in the Builder output that should be enforced next cycle, encode them now. A capability file written after cycle one is worth more than one written before it — it reflects how this Builder actually behaves, not how it theoretically should.
03
Turn the flag ON — or log why not
If the gate passed, the Quality Advocate validates as a real user and recommends flag ON. The PO flips it. If the flag is not being turned ON despite a pass, document the reason in the INTENT_LOG. A flag left permanently in Pending-OFF state after a passed gate is context debt.
04
Answer three questions before cycle two
These are the only retrospective IDF requires after cycle one:
1. Did the Orchestrator decompose without escalating? If no: which part of the intent was ambiguous? → fix in Writing Intent Statements. 2. Did the Guardian gate report give the PO everything needed to decide without asking for more information? If no: what was missing? → add to gate criteria. 3. Did the Builder output require corrections the Guardian did not catch? If yes: what pattern was it? → encode in Capability Library.
These three answers are the only output of the first cycle retrospective. They become direct inputs to System Memory, gate criteria, or the Capability Library. No meeting required. No action items outside those three artefacts.
