The governance model for AI-assisted teams. Agents execute. Humans govern. Everything ships behind a flag.

01 — The shift

Traditional agile was built for human bandwidth. This isn't.

Software teams used to estimate because human execution is slow and variable. Estimation managed that constraint. When agents absorb execution, the constraint disappears — and so should the rituals built around it.

The mental shift

Agents do. Humans govern. Feature flags separate "it's deployed" from "it's live."

02 — What it prioritises

Four values that govern every decision

When the framework has to choose between two goods, these values determine which wins.

Human judgment over automated execution
Usability and clean context over comprehensive documentation
Outcomes over process
Respond to reality over following a plan
03 — Five rules

The rules that govern everything

These are not guidelines. They are the invariants — things the system enforces regardless of team preference, tool choice, or context.

R1 Express outcomes, not instructions. Tell the system what needs to be true for the client. Agents determine how.
R2 Ambiguity stops execution. It never shapes it. If the intent is unclear, the system surfaces the question and waits. It never guesses.
R3 Every client-observable change ships behind a flag. Deployed means hidden in production. Released means a human flipped it on. These are separate decisions.
R4 Context is data plus intent — and it decays. Agents work from written records. Records that aren't maintained drift from reality. Drift produces confident, wrong output.
R5 Measure what reaches the client, not what moves inside the system. Cycle throughput and escape rate are real signals. Commit counts and story points are not.
04 — Roles

Two types of participant. One clear boundary.

Roles in IDF are defined by the decisions they own, not the tasks they perform. One line divides the room: humans govern outcomes, agents execute toward them.

human Govern outcomes
agent Execute toward them
Product Owner Writes intent as client outcomes. The only person who can flip a flag to live.
Orchestrator Decomposes intent into a sequenced task list. Stops if anything is ambiguous.
Craft Engineer Governs how agents write code. Encodes patterns into the capability library.
Builder Executes tasks. Wraps every change in a feature flag set to Pending-OFF.
Quality Advocate Defines what "good" looks like. Validates each deploy as a real user before release.
Guardian Runs all automated checks. Produces the gate report: what was built, signals, pass/fail.
Architects / Tech Leads Set the technical guardrails that agents work within.
No agent holds release authority.
05 — The cycle

One complete unit of work

Every feature follows the same path. Cycles end when the outcome is confirmed — not when the clock runs out.

01
PLAN · Orchestrator
The PO writes an intent statement.
The Orchestrator decomposes it into tasks using the system's written memory. If anything is ambiguous, it stops and asks before proceeding.
02
BUILD · Builder(s)
Agents execute the task list.
Every change goes into production behind a flag set to Pending-OFF. The PO sees nothing yet.
03
AUTO-TEST · Guardian
All automated checks run.
The Guardian produces a gate report in three paragraphs: what was built, the signals, and a pass/fail recommendation.
04
DEPLOY · PO + Craft + QA
Humans review the gate report and approve.
The feature is live on the server — still hidden behind the flag. The Craft Engineer and Quality Advocate check it as real users before the flag moves.
05
MONITOR · PO → flag ON
The PO flips the flag to Live-ON.
Signals are watched. If something is wrong, the flag goes OFF instantly. No rollback required — the code stays deployed, the behaviour is simply hidden again.
THAT'S ONE CYCLE. REPEAT.

Intent → Plan → Build → Gate → Live-ON → Monitor. No sprint planning. No story points. The PO governs the outcome.

FAIL PATH

If auto-test fails: the Guardian reports it, the Orchestrator replans, the Builder fixes it. The PO is not involved until the gate report passes.

06 — What changes

The shift in how your team spends time

IDF does not reduce the amount of human judgment — it relocates it.

Before (Scrum)
  • Writing task lists and story cards
  • Running estimation ceremonies
  • Manual test execution
  • Sprint retrospectives
  • Post-merge rollback planning
After (IDF)
  • Writing intent statements (outcomes, not tasks)
  • Reviewing three-paragraph gate reports
  • Validating as a real user before the flag flips
  • Encoding patterns into the capability library
  • Watching live signals after the flag turns on

The agents absorb execution. The humans absorb judgment. That is the only trade IDF makes.
